# cxpPattenWrite ✍️🛡️

> **别把写代码的脑子直接用来写专利。**
> 
> 这是一个专为工程师和研究员打造的专利交底书“外挂” Skill。它不负责“水字数”，只负责把你的“技术直觉”锻造成“坚不可摧的保护点”。

## ⚠️ 醒醒，你的“技术”还不是“专利”

很多研发人员在写专利时，常陷入三大错觉：

1. **“我的代码跑通了/实验做成了，这就是专利。”** 
   *错！* 专利不保护纯代码，不保护自然规律。专利保护的是**“为了解决具体问题，提出的一套可实施的技术方案”**。
2. **“我刚发了一篇顶会论文，直接改改格式去申请吧。”** 
   *危险！* 论文的目的是**“证明我多牛，并让人信服”**，专利的目的是**“圈地，让别人不能这么干”**。论文越详尽越好，专利则需要在“写清楚”和“保护范围最大化”之间走钢丝。
3. **“我把会议纪要和几页脑图扔给代理师，让他自己悟吧。”** 
   *天真！* 代理师懂法律，但不一定懂你的前沿技术。你不梳理出核心主线，最后写出来的专利保护的可能全是废话。

**`cxpPattenWrite` 就是为了终结这些错觉而生的。**

## 🛑 概念防坑指南：发明 vs 实用新型 vs 论文

在开始用这个 Skill 之前，请先对号入座你的想法到底属于哪一类：

| 维度 | 💡 发明专利 | 🛠️ 实用新型 | 📄 学术论文 |
| --- | --- | --- | --- |
| **核心目的** | 保护**“怎么做”** (流程、方法、算法协同) | 保护**“东西怎么构成”** (结构、形状组合) | **解释并证明**研究发现 |
| **它关注什么**| 步骤、控制逻辑、材料工艺、系统数据流 | 物理形态优化、装置改进、部件连接关系 | 机理分析、实验设计、结果讨论 |
| **写作主线** | 问题 → 方案 → 实施例 → 保护点布局 | 结构组成 → 连接关系 → 部件作用 → 效果 | 背景 → 方法 → 实验 → 结果 → 讨论 |
| **对细节的态度**| 既要能复现，又要留足“变形”的保护空间 | 结构必须具体明确，强调物理上的可实施 | 越详尽越好，无死角证明 |

**一句话总结：** 
- 论文追求 **“讲明白并证明”**。
- 专利追求 **“讲清楚并保护住”**。
- 发明专利重 **逻辑与流程**，实用新型重 **实体与结构**。

## ✨ 这个 Skill 能为你做什么？

`cxpPattenWrite` 沉淀了大量资深专利代理师的“潜规则”，将散乱的材料强制拉入专业框架：

- 🌪️ **乱拳变组合拳**：把聊天记录、零散笔记、会议纪要重塑为有主线的交底书草案。
- 🔍 **榨取保护点**：从你的 Markdown/Docx 草稿中，强行提取出你可能忽略的“保护点”。
- 🏗️ **补全逻辑链**：发现你缺了实施例？缺了系统方案？缺了对比技术？它会列出清单逼你补充，**绝不硬编虚构事实**。
- 🎯 **降维打击**：合并多份 Prompt、多版草稿，按方法、技术、系统三层重组可保护内容。

## 🧠 核心写作哲学：三层剥洋葱与递进法则

我们拒绝平铺直叙，默认采用以下硬核逻辑：

### 1. 铁打的主线
**问题 → 核心方法 → 技术扩展 → 系统整合 → 预期效果**
(脱离这条主线的内容，一律视为废话)

### 2. 保护点“三层剥洋葱”
- **方法层**：保护你的步骤、顺序、条件、输入输出、控制逻辑。
- **技术层**：保护你的材料、结构、工艺、算法、参数、模块关系。
- **系统层**：保护你的系统组成、模块接口、数据流、协同与容错机制。

### 3. 实施例“三步走”
1. **基础实施例**：最小可运行方案（MVP）。
2. **增强实施例**：加料！增加材料、结构、工艺、控制或算法细节。
3. **集成实施例**：升华！打包成完整的系统、流程或工程化落地应用。

## 🚀 极速上手

### 场景一：只有一个粗糙的 Idea (最小输入)
```markdown
专利方向：
要解决的问题：
我的核心思路：
我觉得和现有技术最不一样的地方：
可能的实施方式：
```

### 场景二：缝合怪 (多材料输入)
```markdown
材料 1：思考类（抽问题和方案）
材料 2：整理类（要求按固定章节归类）
材料 3：强化类（要求补实施例、补保护点）
请以材料 1 为主线，吸收 2、3 的优点，整理成一份精简版专利交底书。
```

### 场景三：克制模式 (只整理，不创作)
```markdown
以下内容只允许整理，绝对不允许补写新技术事实。
请按九章结构归类；没有内容的章节标 [待补充]；无法归类的内容统一放到“未归类信息”。
```

### 场景四：压榨模式 (强化扩展)
```markdown
别只给我做排版！基于现有内容，重点做三件事：
1. 提炼核心方法
2. 扩展周边技术方案
3. 组织系统级实施例与保护点，并指出我现在缺什么数据。
```

## ⚙️ 输出形态：按需定制

**默认输出（精简实战派）：**
1. 摘要 -> 2. 技术背景 -> 3. 提出方法 -> 4. 实现方案和步骤 -> 5. 创新点 -> 6. 权利保护

**召唤完全体（当明确要求“九章结构/完整长稿”时）：**
1. 概览与摘要 -> 2. 背景 -> 3. 核心方案 -> 4. 周边配合方案 -> 5. 系统整合方案 -> 6. 权利保护点 -> 7. 具体实施方式 -> 8. 对比现有技术的区别 -> 9. 变形方案与跨域应用
